## 一、产品概要

| 维度 | 结论 |
| --- | --- |
| **产品定位** | 睡眠音响品牌的配套 App，UI 中体现音响品牌形象 |
| **数据来源** | 智能手环/戒指（非音响），通过设备端采集后同步至 App |
| **数据链路** | 手环/戒指 →（离线采集）→ 连接时同步 → App |
| **App 与音响关系** | 无直接通信，仅品牌展示层面关联 |
| **核心数据** | 心率、血氧、睡眠分期（深睡/浅睡/REM/清醒） |
| **目标用户** | 失眠/睡眠障碍人群 |
| **核心场景** | 长期追踪改善——趋势分析 + 改善建议 |
| **目标平台** | iOS + Android |
| **产品风格** | 简单为主 |
| **关键设计点** | 设备非实时连接，需设计**延迟同步**逻辑（断连缓存→连接后自动同步→数据补全） |

## 四、数据同步逻辑

```
┌─────────────┐     离线存储      ┌─────────────┐    蓝牙连接     ┌─────────────┐
│  手环/戒指   │ ───────────────→ │  设备端缓存   │ ─────────────→ │    App      │
│  (采集端)    │   ≤7天睡眠数据    │  (本地存储)   │  增量同步拉取   │  (展示端)    │
└─────────────┘                  └─────────────┘                └─────────────┘
```

| 同步要素 | 策略 |
| --- | --- |
| **同步时机** | ① App 进入前台自动同步 ② 用户首页下拉刷新 ③ 每天早上 8:00 本地通知提醒同步（可关闭） |
| **同步方式** | 增量同步——只拉取上次同步时间之后的新数据 |
| **断点续传** | 中途断开重连后从断点继续 |
| **数据补全** | 设备端最多缓存 7 天数据，连接后一次性回填 |
| **冲突处理** | 以设备时间戳为准，后到的覆盖先到的 |
